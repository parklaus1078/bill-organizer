A cross-platform desktop application built with C++ and Qt for tracking and managing all recurring fixed expenses — including utility bills, OTT subscriptions, LLM API fees, mobile plans, internet bills, and any other periodically charged services.

Subscription & Payment Entry

The app supports a user-defined display currency (e.g., KRW). For entries billed in a foreign currency (e.g., USD, EUR, JPY):

Past payments: Store the converted amount using the exchange rate on the actual payment date (fetched at time of entry or via historical exchange rate API).
Future/upcoming payments: Display the estimated amount nverted at today's exchange rate (fetched live at query time).
Exchange rates are fetched via a public exchange rate API (e.g., ExchangeRate-API, Open Exchange Rates, or the European Central Bank feed).

Dashboard & Summary View

Monthly and annual fixed expense totals, displayed in the user's preferred currency
Breakdown by category (streaming, utilities, cloud/SaaS, telecom, etc.)
Calendar or timeline view of upcoming payment dates
Visual charts (bar/pie) for spending distribution across services

Payment Card Management
A dedicated card wallet to store payment method details linked to subscriptions — card nickname, issuer, last 4 digits (with masking for security), expiry date, and optionally CVC (stored with local encryption).
Alerts & Notifications
Configurable reminders before upcoming billing dates (e.g., notify 3 days in advance via system tray notification).
Data Persistence & Security
All data stored locally using SQLite (via Qt SQL module). Sensitive fields (card details) encrypted at rest usinAES or Qt's built-in security utilities. Optional password/PIN lock on app launch.

Import/export to CSV or JSON
Dark/light theme toggle
Currency auto-update scheduler (refresh exchange rates daily in background)

In-app viewer for images and PDFs (via Qt's QPdfDocument or embedded WebEngine for HTML)
Thumbnail preview in the payment history list
Full-screen view with zoom support for images

Invoice files stored in a local structured folder (/invoices/{year}/{month}/{service_name}/)

## Interpretation Notes

### User Authentication Decision
**No user authentication feature included.** This is a desktop application with local-only data storage (SQLite). Multiple user accounts or cloud sync are not mentioned in requirements, so auth is unnecessary. If multi-user support is needed later, it can be added as a separate feature.

### Payment Card Management as Foundation
Payment card management (PLAN-001) is separated from subscription management (PLAN-002) because:
- Cards are reusable entities linked to multiple subscriptions
- Card CRUD operations are simpler and can be completed independently
- Subscription feature depends on having payment cards available for linking

### Currency System Integration
Currency & Exchange Rate System (PLAN-003) is separated from Subscription Management (PLAN-002) to:
- Allow subscription CRUD to work independently (store original currency, no conversion initially)
- Keep exchange rate API integration as a distinct concern
- Enable future currency features (manual rates, offline mode) without affecting core subscription logic

### Dashboard Dependencies
Dashboard (PLAN-004) depends on both PLAN-002 and PLAN-003 because:
- Needs subscription data for aggregations
- Requires currency conversion to display all amounts in user's preferred currency

### Invoice Storage Complexity
Invoice & Receipt Storage (PLAN-006) is marked as "Large" complexity because:
- Multiple file format support (image, PDF, HTML)
- File system organization and management
- In-app viewer implementation with zoom/pan
- Thumbnail generation
- Security considerations (file validation, size limits)

## Reviewer Notes

### Open Questions

1. **Multi-User Support**: Is this application intended for single-user only, or should we plan for multi-user scenarios (e.g., family sharing)? Current plan assumes single-user.

2. **Cloud Sync**: Requirements mention "local storage" but don't explicitly rule out cloud sync. Should we plan for cloud backup/sync in future, or is this strictly offline-only?

3. **Exchange Rate API Selection**: Which exchange rate API should be prioritized? ExchangeRate-API requires API key; ECB is free but EUR-based. Needs decision before PLAN-003 implementation.

4. **OCR Priority**: PLAN-007 marked as "Low" priority, but invoice storage (PLAN-006) is "Medium". Should OCR be implemented immediately after PLAN-006, or can it be deferred to later releases?

### Arbitrary Decisions Made

1. **PLAN-001 First**: Payment Card Management placed before Subscription Management because cards are dependencies. Could be implemented in parallel if needed.

2. **PLAN-005 vs PLAN-006 Order**: Both have same dependency (PLAN-002) and can be implemented in parallel. Notification system slightly simpler, so placed first for sequential implementation.

3. **Complexity Ratings**: "Large" assigned to features with multiple sub-systems (file handling + viewer + storage for PLAN-006; multiple chart types + calendar + aggregations for PLAN-004).

### Features That May Need Separation/Merging

**Consider Merging PLAN-007 into PLAN-006**: OCR is tightly coupled with invoice processing. Might be more efficient to implement as optional sub-feature of PLAN-006 rather than standalone ticket.

**Consider Separating Calendar from PLAN-004**: Calendar/timeline view could be standalone feature if it becomes complex (e.g., interactive calendar with click-to-edit, drag-and-drop rescheduling).

**Category Management**: Currently included in PLAN-002 but could be separated if category features expand (hierarchical categories, custom icons, spending limits per category).
